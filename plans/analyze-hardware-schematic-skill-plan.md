# 创建通用 Skill：`analyze-hardware-schematic`

## Summary
- 这个 skill 的定位是通用的硬件原理图分析 skill，不绑定当前仓库，也不限定只能做“旧文档纠偏”。
- skill 的核心任务是：根据用户提供的总体说明、模块划分、模块截图资料和 BOM 文档，按模块分析硬件设计，识别模块内和模块间的实际链接关系，输出模块级文档、总体模块间链接关系文档，以及项目级待确认/冲突清单。
- skill 必须同时支持两种工作方式：`从零生成文档` 和 `修订已有文档`。
- skill 必须遵守保守证据原则：不清楚的链接关系不猜测，文档中显式标注待确认项；当不确定信息过多且已经影响主链路判断时，停止继续下结论，等待用户确认。

## Skill 目标
- 让 Codex 能在不直接通读整个原理图工程文件的情况下，基于模块化截图资料完成硬件设计分析。
- 把“总体设计理解”和“模块局部链路分析”结合起来，而不是只对局部截图做描述。
- 产出结构稳定、证据边界清晰、便于后续持续修订的硬件设计文档。

## 必需输入
以下输入都应在 `SKILL.md` 中定义为必需项，而不是可选项。

### 1. 总体和模块间描述文档
- 需要包含整机或板级功能说明。
- 需要列出主要模块划分，例如：电源模块、模拟信号输入前端、模拟信号输出、I2C/SPI/UART 通信、MCU 等。
- 需要说明各模块与 MCU、外部 I/O、其他模块之间的连接关系。
- 需要尽量写清通过哪些信号名、网络名或引脚相连。

### 2. 模块化截图资料
- 每个模块单独使用一个文件夹存放局部原理图截图。
- 截图文件名应能体现功能和链接顺序，例如：
  - `01_ACDC电源-24V.png`
  - `02_DC电源24V-5V.png`
- 截图顺序应尽量反映实际分析路径或电路链路方向。

### 3. BOM 文档
- 标准文件名为 `bom.md`。
- 兼容历史别名 `boom.md`。
- 如果用户显式指定了其他 BOM 文件路径，则以用户指定路径为准。
- 如果 `bom.md` 和 `boom.md` 同时存在，优先使用 `bom.md`，并记录存在历史别名兼容。
- 文件开头必须先给出 Markdown 表格。
- 表格最低字段要求为：
  - `位号 / RefDes`
  - `器件型号`
  - `参数`
  - `功能/器件类别名`
  - `所属模块`
- 如项目使用页码或 Sheet 标识，可附加：
  - `原理图编号 / Sheet ID`
- 表格后必须有独立章节补充关键器件的技术参数、用途、替代料或原图 Comment 等信息。

### 4. 现有文档
- 现有模块文档和总体关系文档不是必需输入。
- 如果用户提供了现有文档，skill 应进入“修订模式”。
- 如果用户没有提供现有文档，skill 应进入“从零生成模式”。

## 推荐输入
以下输入不是强制项，但应在计划中列为推荐输入。缺少这些资料时，skill 仍可工作，但应明确指出分析闭环能力会下降。

### 1. `module-index.md`
- 推荐作为项目级模块索引。
- 每个模块至少包含：
  - 文件夹路径
  - 模块描述
  - 上游输入
  - 下游输出
  - 相关截图文件名列表

### 2. `mcu-pin-map.md`
- 说明 MCU 关键引脚与各模块的对应关系。

### 3. `interface-map.md`
- 说明外部接口、连接器、端子与内部模块的映射关系。

### 4. `schematic-page-map.md`
- 说明模块与原理图页码、Sheet 名、分页结构之间的对应关系。

### 5. 截图补充说明文本
- 对单张截图补充上下游说明、链路方向或截图边界说明。

### 6. 整份原理图 PDF 导出件
- 仅作为辅助证据。
- 用于定点补查页码、Sheet 名、跨页标签、接口页对应关系。
- 不应替代模块化截图分析流程。

## 输入验收与图片质量要求
skill 在正式分析前必须先执行一次输入验收，而不是拿到资料后直接开始写结论。

### 输入验收项
- 必需输入是否齐全。
- 模块目录和截图命名是否有顺序和功能语义。
- BOM 中的 `位号 / RefDes` 是否能与截图中的器件位号建立映射。
- 总体说明中的模块划分是否能与截图目录基本对应。
- 推荐输入是否存在，缺失项是否会影响关键链路分析。

### 图片质量要求
- 网络名、引脚名、位号在正常缩放下必须可读。
- 关键连接不能被裁掉。
- 一张图应尽量只覆盖一个连续链路片段，不要把多个无关链路混在一张图里。
- 文件名必须按分析顺序编号。
- 如果图片模糊、压缩严重、关键信息遮挡，skill 应先输出“输入质量问题清单”，而不是直接进入结论阶段。

## 证据规则
以下规则应作为 `SKILL.md` 中的强约束，而不是建议性文字。

### 证据优先级
- 对连接关系和电路拓扑的判断：
  - 截图中的器件位号、网络名、引脚名、跨图标签、连接关系和电路拓扑是一级证据。
  - `module-index.md`、`mcu-pin-map.md`、`interface-map.md`、`schematic-page-map.md` 属于辅助证据。
  - 整份原理图 PDF 只可用于经用户允许后的定点补证，不得替代截图主证据。
- 对器件身份和参数的判断：
  - BOM 文档是器件型号、参数、类别信息的主证据。
  - 截图中的位号是 BOM 与电路之间的连接桥梁。

### 冲突处理
- 如果截图与总体说明冲突，应以截图中的实际连接为准，并显式记录冲突。
- 如果截图与 BOM 冲突，应先确认是否是截图标注错误、BOM 过期或替代料信息未同步；在未确认前不得写死结论。
- 如果辅助证据之间彼此冲突，但截图未能闭环，应把该关系标记为“冲突”或“待确认”，并暂停相关结论。

### 不确定性处理
- 看不清、看不全、没有跨图闭合证据的链接关系，不得擅自补全。
- 允许做有限推断，但必须显式标记为“推断”，并说明推断依据。
- 任何影响主链路判断的不确定关系，都应在文档中单列为“待确认项”。

### 禁止行为
- 不得在用户未授权时主动查看完整原理图工程文件来补证据。
- 不得把 BOM 中的型号信息反向当作连接关系证据。
- 不得为了让文档“完整”而隐式省略不确定点。

## 工作流程
`SKILL.md` 中的流程应固定为以下顺序。

### 0. 输入验收
- 先检查输入是否满足最小要求。
- 如果输入不达标，先输出问题清单和补充建议，不进入正式分析。

### 1. 建立总体模块地图
- 先阅读总体和模块间描述文档。
- 如存在 `module-index.md`，优先利用它建立模块地图。
- 识别主要模块、模块边界、模块功能以及与 MCU / I/O 的连接关系。
- 形成初始的模块关系框架，但此时不把未被截图验证的关系写成结论。

### 2. 识别关键优先链路
- 优先识别高风险、高价值链路，例如：
  - 电源链路
  - 复位与时钟链路
  - 基准电压、参考地、模拟前端参考链路
  - MCU 主通信总线
  - 关键输入输出接口链路
- 这些链路若证据不足，应优先触发暂停或待确认，而不是继续扩散分析。

### 3. 逐模块分析截图资料
- 进入每个模块文件夹，按截图命名顺序分析子模块链路。
- 利用截图中的网络名、位号、跨图标签和信号方向，建立该模块内部的子链路关系。
- 如有现有模块文档，逐段核对其是否符合截图描述；如无现有文档，则直接生成。

### 4. 结合 BOM 收敛器件信息
- 对模块中的关键器件，从 BOM 中提取型号、关键参数、器件功能和所属模块信息。
- 关键器件信息应集中放在对应子模块开头，而不是散落在全文各处。
- 当截图标注与 BOM 冲突时，必须显式记录冲突并暂停写死结论。

### 5. 输出模块级文档
- 每个模块文档按子模块组织。
- 每个二级标题代表一个子模块。
- 每个子模块开头先列“关键器件”，再写链路分析、关键关系表、确认结论和待确认项。

### 6. 输出总体模块间链接关系文档
- 只有在各模块主要链路已基本分析完成后，才生成总体模块关系文档。
- 总体文档只汇总已经在模块分析中获得证据支持的模块关系。
- 未闭合的跨模块关系必须保留为“待确认”，不能在总体文档中伪装成闭环。

### 7. 输出项目级待确认/冲突清单
- 汇总跨模块未闭合关系、BOM 冲突、命名不一致、截图缺失和需要用户补证的项目。
- 该清单应作为后续迭代输入，而不是只散落在各模块文档中。

## 停止条件
skill 必须带有明确的停止策略。

### 可以继续的情况
- 存在少量局部不确定信息，但不影响该模块主链路判断。
- 某些器件参数尚未确认，但截图中的连接关系已经足够支撑当前分析。

### 必须暂停并等待用户确认的情况
- 某个关键链接关系不明确，已经影响该子模块或该模块主链路判断。
- 电源、复位、时钟、参考链路、MCU 主总线等关键链路证据不足。
- BOM 与截图存在明显冲突，且冲突会影响器件功能或链路解释。
- 总体和模块间描述文档与截图实际连接差异较大，无法判断应以哪种边界继续分析。

### 必须停止本轮工作的情况
- 大量关键链接关系不清楚，已经无法可靠建立模块主链路。
- 跨模块关系缺失过多，无法生成可信的总体模块间链接关系文档。
- 输入资料本身不满足最小要求，例如缺少总体说明、模块截图结构混乱、BOM 信息严重不足、图片不可读。

## 输出要求

### 模块级文档
- 以模块为单位输出，每个模块一份文档。
- 以子模块为主要组织结构，不以器件清单堆砌正文。
- 每个子模块应包含：
  - `关键器件`
  - `电路关系分析`
  - `关键关系表`
  - `当前可确认的内容`
  - `待确认项`
  - `冲突项`

### 关键关系表
- 每个子模块中的关键关系应尽量结构化记录，至少包含：
  - `关系描述`
  - `证据截图文件名`
  - `关键位号 / RefDes`
  - `关键网络名`
  - `是否闭环`
  - `是否跨模块`
  - `置信度`

### 置信度等级
- 置信度标签统一使用：
  - `已确认`
  - `推断`
  - `待确认`
  - `冲突`

### 总体模块间链接关系文档
- 应独立于模块级文档单独输出。
- 应覆盖：
  - 模块清单和功能角色
  - 电源链路
  - 参考电位/基准链路
  - 关键信号输入输出链路
  - MCU 与各模块的交互链路
  - 当前未闭合或待确认的跨模块关系

### 项目级待确认/冲突清单
- 应独立输出，或者作为总体模块关系文档的固定附录。
- 至少包含以下字段：
  - `问题项`
  - `截图证据`
  - `BOM/总体文档/辅助证据`
  - `影响`
  - `当前处理`
  - `需要用户确认的内容`

### 从零生成与修订两种模式
- 如果已有文档存在，输出应以“修订和纠偏”为主。
- 修订模式下，应显式核对旧结论是否有截图或 BOM 证据支撑。
- 如果已有文档不存在，输出应以“按统一结构新建文档”为主。
- 两种模式都必须遵守同一套证据和停止规则。

## `SKILL.md` 编写要求

### Frontmatter
建议使用以下方向的 frontmatter，确保 skill 会在“通用硬件原理图分析”场景被触发：

```md
---
name: analyze-hardware-schematic
description: Analyze modular hardware schematic materials to create or revise hardware design documentation. Use when the user provides an overall architecture description, per-module schematic screenshots, and a BOM document such as bom.md, and the task is to understand module internals, confirm inter-module links, identify unresolved connections, and produce module-level documents plus an overall module-link document without guessing unclear relationships.
---
```

### 正文结构
`SKILL.md` 正文建议固定为以下几段：
- `When to Use`
- `Required Inputs`
- `Recommended Inputs`
- `Input Validation`
- `Evidence Rules`
- `Workflow`
- `Stop Conditions`
- `Output Requirements`
- `Reference Files to Read`

正文只保留核心流程和约束，具体模板与检查项下沉到 `references/`。

## `references/` 内容规划

### `input-requirements.md`
- 解释输入资料的最小要求和推荐要求。
- 给出总体说明文档、模块截图目录、`module-index.md` 和 BOM 的推荐组织方式。
- 明确截图命名顺序和图片质量要求。

### `module-doc-template.md`
- 约定模块文档的统一结构。
- 明确每个子模块先写关键器件，再写链路分析和关键关系表。

### `inter-module-links-template.md`
- 约定总体模块关系文档结构。
- 强制保留“待确认跨模块关系”一节。

### `project-open-issues-template.md`
- 约定项目级待确认/冲突清单的结构和字段。

### `evidence-rules.md`
- 解释证据等级和置信度标签。
- 规定截图、BOM、总体说明和辅助证据冲突时的处理方式。

### `stop-rules.md`
- 细化什么时候继续、什么时候暂停、什么时候停止本轮工作。
- 明确关键链路优先暂停原则。

### `review-checklist.md`
- 检查是否有无依据的补全。
- 检查是否把器件信息散落在正文各处。
- 检查是否为每条关键结论附了证据引用。
- 检查总体关系文档是否只基于已确认模块分析结果。
- 检查项目级待确认/冲突清单是否完整。

## 验证方式
实现 skill 后，应至少用以下场景验证：

1. 用户提供总体说明、分模块截图和 BOM，但没有旧文档。
   - skill 应能从零生成模块文档、总体关系文档和项目级待确认/冲突清单。

2. 用户同时提供旧文档。
   - skill 应优先做纠偏和修订，而不是完全重写。

3. 截图中存在关键跨图链路缺失。
   - skill 应把缺失关系列为待确认，而不是猜测补齐。

4. BOM 与截图存在冲突。
   - skill 应显式指出冲突，并在必要时停止继续下结论。

5. 用户同时提供 `bom.md` 和 `boom.md`。
   - skill 应优先使用 `bom.md`，并记录历史别名兼容。

6. 图片模糊或关键连接被裁掉。
   - skill 应先输出输入质量问题清单，而不是继续生成正式结论。

通过标准：
- skill 能自然触发到“硬件原理图分析”场景，而不是仅限局部截图改文档。
- skill 会先检查输入是否满足要求。
- skill 会稳定遵守“不清楚不猜、冲突即标注、必要时停止”的规则。
- skill 会稳定给出证据引用、置信度和冲突记录。
- skill 能稳定支持“从零生成”和“修订已有文档”两种模式。

- 后续实施时，先创建 skill 目录和 `SKILL.md` 骨架，再补 `references/`。
- 如果后续发现这个通用版还需要适配某类固定仓库结构，再单独派生仓库专用版，而不是污染通用版。
